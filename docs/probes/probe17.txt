Core finding

- Confirmed:
  - The frontend contains hardcoded production backend domains and hardcoded Azure blob storage domains.
- Most concerning files:
  - `src/hooks/use-image-validation.ts`
  - `src/pages/PromoAdminPage.tsx`
  - `src/lib/generation-history-api.ts`
  - `src/lib/generation-enrichment.ts`
  - `src/lib/azure-utils.ts`
  - `src/lib/model-library.ts`
- Practical meaning:
  - Parts of the app cannot cleanly move between local, staging, and production.
  - Backend/storage rotation can require frontend source changes and redeploys.
  - Photo validation, history enrichment, promo admin, and model/blob loading are the biggest portability risks.

Recommended cleanup target for later

- Create one environment/backend URL policy:
  - protected backend calls use relative `/api/...` plus `authenticatedFetch`, or a single configured API base.
  - direct storage URLs are returned by backend or generated through one configurable asset helper.
  - hardcoded production domain stays only in SEO/legal/marketing places where that is intentional.
  - internal email gates move to backend/admin config when the temporary CAD gates become product features.
