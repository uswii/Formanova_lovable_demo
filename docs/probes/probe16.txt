- Important behavior of `ChunkErrorBoundary`:
  - It catches only chunk-load errors.
  - `isChunkError(error)` returns true for messages like:
    - dynamically imported module failure
    - failed to fetch
    - loading chunk
    - importing module script failed
  - For chunk errors:
    - if generation is active:
      - shows a modal telling user generation is still processing
    - if generation is not active:
      - attempts guarded auto-reload after 1.5s
  - For non-chunk errors:
    - `getDerivedStateFromError` does:
      - `throw error`
    - meaning normal component errors are deliberately not contained by this boundary.

- What happens if `UnifiedStudio.tsx` throws?
  - If the throw is a chunk-load/dynamic import error:
    - `ChunkErrorBoundary` catches it.
    - blast radius:
      - route/main content is replaced by the boundary UI.
      - global `Header` remains outside the boundary.
      - if generation is active, user sees the generation-safe reload modal.
  - If the throw is a normal render/lifecycle exception:
    - `ChunkErrorBoundary` rethrows it.
    - there is no outer React error boundary in `main.tsx`.
    - blast radius:
      - likely the whole React app crashes/unmounts for the user.
      - not just the studio panel.
  - If the throw happens inside an async event handler or promise:
    - React error boundaries generally do not catch async errors.
    - behavior depends on local `try/catch`.
    - `UnifiedStudio` catches many generation-flow errors locally and maps them to UI state, but uncaught async errors can still surface globally.

- Does `UnifiedStudio` have a local panel/page boundary?
  - no.
  - no boundary wrapping:
    - upload step
    - model selection
    - generation panel
    - result panel
    - feedback modal
  - Therefore a render-time crash in a child of `UnifiedStudio` is not contained to one studio section.

- Does CAD have local boundaries?
  - `TextToCAD.tsx`
    - no real local React error boundary found.
  - `CADCanvas.tsx`
    - no real local React error boundary found.
  - `InteractiveRing.tsx`
    - has a local `ErrorBoundary3D` function, but it is not a proper React error boundary.
  - CAD canvas/runtime errors are mostly handled manually through local loading/error state and try/catch, not through React boundary containment.

- Current blast radius summary:
  - Chunk-load failure inside a routed page:
    - contained to the route/main area by `ChunkErrorBoundary`.
  - Normal render/lifecycle crash inside a routed page:
    - not contained.
    - app-level crash risk.
  - Normal render/lifecycle crash in `Header`, providers, router helpers, decorations, or update banner:
    - not contained by any active boundary.
    - app-level crash risk.
  - Async handler/promise errors:
    - not caught by React boundaries unless local `try/catch` handles them.

- Risk score:
  - Active real error boundaries:
    - 1
  - Active route/page-local workflow boundaries:
    - 0
  - High-risk god pages without local boundary:
    - `UnifiedStudio.tsx`
    - `TextToCAD.tsx`
  - High-risk renderer without real local boundary:
    - `CADCanvas.tsx`
  - Boundary coverage quality:
    - low for normal app errors
    - medium only for chunk-load/deploy errors

- Core finding:
  - FormaNova has a chunk-load recovery boundary, not a general app safety boundary.
  - If `UnifiedStudio.tsx` has a normal render crash, it does not merely break the studio panel.
  - Because `ChunkErrorBoundary` rethrows non-chunk errors and `main.tsx` does not wrap `<App />`, a normal unhandled component error can tear down the whole React app for the user.
  - For refactoring, the missing boundary is not "one more global wrapper" only.
  - The safer target is layered containment:
    - app-level catch-all boundary around `<App />`
    - route-level boundary around each major tool route
    - local boundary around CAD/WebGL rendering
    - possibly section-level boundaries inside `UnifiedStudio` after it is split into smaller components
