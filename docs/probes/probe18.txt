Probe 18: Raw `fetch()` calls hitting `/api/` in `src/lib/` and `src/pages/`

Prompt:
- Find every raw `fetch()` call in `src/lib/` and `src/pages/` that hits `/api/`.
- For each:
  - file and line
  - whether it manually gets token via `getStoredToken()`
  - what happens on `401`
- Goal:
  - create a checklist to migrate all calls to `authenticatedFetch`.

Scope:
- Searched `src/lib` and `src/pages`.
- Included raw `fetch()` calls that directly target `/api/...` or use a local constant equal to `/api`.
- Also noted dynamic raw downloads that could become risky if passed `/api/artifacts/...`, but kept them separate because they do not statically hit `/api`.

Commands used:
- `rg -n "fetch\(" src/lib src/pages`
- `rg -n "getStoredToken|401|Unauthorized|status === 401|status\s*==\s*401" src/lib src/pages`
- `rg -n "downloadAuthImage\(|/api/|API_BASE|ADMIN_GENERATIONS_BASE|BASE_URL" src/pages/AdminFeedbackPage.tsx src/lib/admin-generations-api.ts src/lib/photoshoot-api.ts src/pages/LinkAccount.tsx src/pages/TextToCAD.tsx src/lib/pipeline-api.ts`

Core finding

- The most important migration target is `src/lib/photoshoot-api.ts`.
- Photo/product-shot generation still uses the older raw-fetch-plus-manual-token pattern for every protected workflow lifecycle call:
  - start,
  - status,
  - result,
  - product-shot start.
- On 401, these calls currently look like workflow/generation failures instead of session expiry.
- `authenticatedFetch` would centralize the expected behavior:
  - clear stored token/user,
  - dispatch auth change,
  - redirect to login with destination preserved,
  - throw `AuthExpiredError`.
