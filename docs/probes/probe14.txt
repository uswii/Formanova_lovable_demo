- Highest auth bypass risks:
  - `src/lib/photoshoot-api.ts`
    - live photo/product-shot workflow start, status, and result all bypass `authenticatedFetch`.
  - `src/hooks/use-image-validation.ts`
    - live classification workflow start, status, and result all bypass `authenticatedFetch` and use hardcoded `https://formanova.ai`.
  - `src/lib/admin-generations-api.ts`
    - live admin generations list/detail bypass centralized 401 handling.
  - `src/pages/LinkAccount.tsx`
    - live protected `/api/agent/link/complete` call bypasses centralized 401 handling.

- Auth behavior consequence:
  - For calls that use `authenticatedFetch`:
    - missing token or 401 can trigger centralized auth-expiry handling.
    - token/user cleanup and redirect behavior are centralized.
  - For raw fetch + manual token calls:
    - missing token behavior is local and inconsistent.
    - 401 responses are treated as ordinary errors.
    - token/user cleanup does not happen centrally.
    - session expiry can appear as generation failure, validation fallback, admin error, or link failure instead of a clean re-login flow.

- Core finding:
  - The bypass pattern is widespread enough to matter, but not random everywhere.
  - It is concentrated in exactly the risky backend workflow paths:
    - photo generation
    - image validation
    - admin generation inspection
    - account linking
  - CAD generation is more consistent because `TextToCAD.tsx` uses `authenticatedFetch` for `/api/run`, `/api/status`, and `/api/result`.
  - Photo/on-model still has the older direct-token pattern.
  - The standard should be:
    - protected `/api`, `/billing`, `/assets`, `/admin`, and artifact proxy URLs use `authenticatedFetch`
    - raw `fetch` is allowed only for:
      - auth bootstrap endpoints
      - static public assets
      - `data:` / `blob:` URLs
      - direct public download URLs after an explicit protected-vs-public check
      - the internals of `authenticatedFetch` itself
